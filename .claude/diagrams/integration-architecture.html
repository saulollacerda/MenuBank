Opção de repositório — monorepo (recomendado para agora)

Monorepo Maven multi-module Recomendado

  • Um único repositório com múltiplos módulos
  • shared-contracts é só um módulo local — sem publicar JAR
  • CI/CD único — um pipeline constrói tudo
  • Refactor cross-service em um único commit
  • Quando extrair: mover a pasta para repo próprio depois

Repositórios separados

  • Cada serviço em seu próprio repo
  • shared-contracts publicado no GitHub Packages
  • CI/CD independente por serviço
  • Mudança de contrato exige bump de versão
  • Complexidade operacional maior agora
Estratégia: começar como monorepo Maven multi-module. Quando o time crescer ou os serviços precisarem de ciclos de deploy independentes, mover cada módulo para seu próprio repo é trivial — a estrutura interna já estará certa.

anota-ai-adapter/pom.xml — novo serviço

<project>
  <parent>
    <groupId>com.menubank</groupId>
    <artifactId>menubank-parent</artifactId>
    <version>1.0.0-SNAPSHOT</version>
  </parent>

  <artifactId>anota-ai-adapter</artifactId>

  <dependencies>
    <!-- contrato compartilhado com o core -->
    <dependency>
      <groupId>com.menubank</groupId>
      <artifactId>shared-contracts</artifactId>
      <version>${contracts.version}</version>
    </dependency>

    <!-- web (webhook + sync endpoints) -->
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-webmvc</artifactId>
    </dependency>

    <!-- RabbitMQ -->
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-amqp</artifactId>
    </dependency>

    <!-- validação e actuator -->
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-validation</artifactId>
    </dependency>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-actuator</artifactId>
    </dependency>

    <!-- sem JPA, sem banco próprio -->

    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-test</artifactId>
      <scope>test</scope>
    </dependency>
  </dependencies>

  <build>
    <plugins>
      <plugin>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-maven-plugin</artifactId>
      </plugin>
    </plugins>
  </build>
</project>
Sem JPA, sem banco: o adapter é stateless. Não tem spring-boot-starter-data-jpa, não tem DataSource. Só web + AMQP + shared-contracts.

application.yml — anota-ai-adapter

server:
  port: 8081   # core roda em 8080

spring:
  application:
    name: anota-ai-adapter

  rabbitmq:
    host: ${RABBITMQ_HOST:localhost}
    port: ${RABBITMQ_PORT:5672}
    username: ${RABBITMQ_USER:menubank}
    password: ${RABBITMQ_PASS:menubank}

anota-ai:
  base-url-orders: https://api-parceiros.anota.ai/partnerauth
  base-url-catalog: https://api-menu.anota.ai/partnerauth

  # configuração por merchant
  merchants:
    - id: ${MERCHANT_1_ID}             # UUID do merchant no MenuBank
      api-key: ${MERCHANT_1_API_KEY}     # token Bearer para Anota.AI API
      webhook-token: ${MERCHANT_1_WHK}  # token externo configurado no portal
    - id: ${MERCHANT_2_ID}
      api-key: ${MERCHANT_2_API_KEY}
      webhook-token: ${MERCHANT_2_WHK}

  polling:
    enabled: true
    cron: "0 */5 * * * *"   # a cada 5 minutos

  catalog-sync:
    cron: "0 0 3 * * *"     # toda madrugada às 3h

rabbitmq-config:
  exchanges:
    orders: menubank.orders
    catalog: menubank.catalog
    requests: menubank.requests
  queues:
    orders-anotaai: orders.anotaai
    catalog-anotaai: catalog.anotaai
    sync-requested: catalog.sync.requested
Configuração de merchants por variável de ambiente: cada merchant tem seu próprio conjunto de variáveis. Em produção (Railway), cada variável é injetada no deploy. Em desenvolvimento, usa um .env local. Sem banco próprio no adapter.

Management UI: http://localhost:15672 — interface visual do RabbitMQ para inspecionar filas, mensagens, DLQ e replay de mensagens com erro. Essencial para debug durante o desenvolvimento.

Roteiro de implementação — ordem recomendada

PASSO 1

Parent POM + shared-contracts

Criar o pom.xml raiz. Criar o módulo shared-contracts com os records dos eventos. Fazer o backend/pom.xml apontar para o parent.

PASSO 2

RabbitMQ local

Adicionar o RabbitMQ no docker-compose.yaml. Verificar que sobe e que o Management UI está acessível em :15672.

PASSO 3

Esqueleto do adapter

Criar a pasta anota-ai-adapter/ com o pom.xml, Application.java e application.yml. Confirmar que sobe na porta 8081.

PASSO 4

Mover client + DTOs

Mover AnotaAIClient, AnotaAIOrderDetailResponse, AnotaAIOrderListResponse, AnotaAICatalogResponse do core para o adapter.

PASSO 5

Criar mappers (TDD)

Escrever testes do AnotaAIOrderMapper e AnotaAICatalogMapper. Implementar. Lógica de formato saída do AnotaAIOrderImportService.

PASSO 6

Webhook + publisher

Criar AnotaAIWebhookController e RabbitMQConfig no adapter. Publicar OrderReceivedEvent na fila. Testar com Postman.

PASSO 7

Consumer no core

Criar OrderConsumer @RabbitListener e CatalogConsumer no core. Adaptar assinatura do AnotaAIOrderImportService.

PASSO 8

Polling + catalog sync

Mover o PollingWorker e o CatalogSyncController para o adapter. Criar o CatalogSyncRequestListener.

TDD em todos os passos: cada mapper, controller e consumer tem teste antes da implementação. Os mappers são os mais críticos — são lógica pura, sem dependências externas, ideais para TDD.
